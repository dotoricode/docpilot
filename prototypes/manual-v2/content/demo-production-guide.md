# DocPilot 데모 제작 가이드

## 증거 계약

모든 기능 설명은 실제 DocPilot 화면에서 다음 세 상태를 확인할 수 있어야 합니다.

1. `entry`: 기능을 시작하는 버튼, 메뉴, 탭 또는 클릭 영역과 화면 안 위치
2. `opened`: 진입점을 누른 직후 열린 메뉴, 패널, 보기 또는 배치 미리보기
3. `result`: 기능이 실제로 완료된 뒤 사용자가 확인할 수 있는 결과

정적인 기능은 위 상태를 1–3장의 이미지로 보여 줍니다. 공간 관계, 연속 동작, 실제 입력 속도 또는 로딩 시간이 핵심이면 영상으로 보여 줍니다. 이미지와 영상 모두 실제 Electron 앱과 합성 fixture에서 만들어야 합니다.

기능별 매체와 세 상태의 기계 판독 가능한 기준은 `feature-media-contract.json`을 사용합니다.

## 재생 규칙

- 영상이 화면의 60% 이상 들어오면 처음부터 자동 재생합니다.
- 화면 밖으로 나가면 일시 정지합니다.
- 마지막 결과 프레임은 약 3초 유지하고, 영상이 보이는 동안 계속 반복합니다.
- 다른 문서 경로로 이동하면 이전 영상의 source, poster, observer, 재생 위치와 제어 상태를 모두 버립니다.
- `prefers-reduced-motion` 환경에서는 자동 재생하지 않고 실제 완료 상태 포스터와 수동 제어를 제공합니다.
- 각 영상의 일시 정지, 재생, 처음부터 다시 보기 제어는 유지합니다.

## 사람 속도와 카메라

- 타이핑은 `fill()`로 한 번에 채우지 않고 문자 단위로 입력하며 단어와 명령 사이에 자연스러운 짧은 멈춤을 둡니다.
- 포인터는 easing을 적용한 중간 지점을 따라 움직입니다. 버튼 근처로 순간 이동하지 않습니다.
- 클릭 전에 사용자가 버튼과 위치를 확인할 시간을 둡니다.
- 메뉴, 드래그, 배치 전환은 실제 속도를 유지합니다. 로딩과 입력 없는 대기도 성능 증거 영상에서는 가속하지 않습니다.
- 작은 대상의 레이블을 읽어야 할 때만 1.25–1.55배 확대합니다.
- 클릭 위치와 결과가 동시에 보이지 않으면 확대를 늦추거나 전체 화면으로 먼저 돌아옵니다.
- 한 영상에는 한 기능 이야기를 담되, Diff·Pane·Claude 전달처럼 승인된 다단계 흐름은 입력부터 결과까지 자르지 않습니다.
- 영상 길이는 시나리오 완결성으로 결정합니다. 짧은 길이를 위해 행동 속도를 높이지 않습니다.

## 캡처와 개인정보

- 실제 DocPilot Electron 앱을 1440×900, 30fps로 캡처합니다.
- 격리된 임시 Git 프로젝트와 합성 문서만 사용합니다.
- 회사명, 고객명, 사용자 경로, 저장소 비밀, 인증 정보, 알림과 관련 없는 창을 캡처하지 않습니다.
- AsciiDoc fixture는 실제 `manual.adoc`처럼 장과 절, 표, 코드, 목록이 충분한 합성 매뉴얼이어야 합니다.
- AsciiDoc 데모는 앱을 새로 실행한 뒤 최초 Preview 변환과 같은 파일의 캐시 재열기를 모두 실제 시간으로 보여 줍니다.
- Claude 데모는 설치된 실제 CLI를 실행하되 합성 문서만 사용합니다. DocPilot Preview에서 클릭해 만든 `파일명 + 줄 번호 + 본문`을 붙여넣고 Enter로 제출되는 장면까지만 완료 기준으로 삼습니다.

## 재현 절차

```sh
node scripts/capture-manual-v2-demos.js all
node scripts/process-manual-v2-demos.mjs
```

원본과 단계 메타데이터는 `.tink/current/artifacts/demo-raw/`에 둡니다. 공개 매뉴얼에는 검수를 통과한 결과만 둡니다.

- MP4: 1440×900, 30fps, H.264 master
- WebM: 960px 폭, VP9 문서용 영상
- GIF: 필요한 경우에만 만드는 공유용 대체 파일
- JPG/PNG: 실제 완료 상태 포스터와 기능 증거 이미지
- Contact sheet: `.tink/current/artifacts/demo-processed/`의 전체 단계 검토 이미지

## 핵심 다단계 시나리오

### 탭과 Pane 배치

1. 두 문서 탭을 열고 문서 Pane을 왼쪽, 오른쪽, 위, 아래에 배치합니다.
2. Terminal 버튼의 위치를 보여 주고 Terminal Pane을 엽니다.
3. Terminal Pane도 문서 기준 왼쪽, 오른쪽, 위, 아래에 배치합니다.
4. 드롭 미리보기와 실제 결과 방향이 같은지 전체 화면에서 확인합니다.
5. 기존 `terminal/layout` 설명은 이 흐름으로 통합하되 기존 URL은 호환 경로로 유지합니다.

### Diff

1. Source에서 실제 문장을 사람이 따라갈 수 있는 속도로 수정합니다.
2. Preview 또는 Source에서 Diff를 켭니다.
3. 추가·삭제 결과와 Summary를 확인합니다.
4. 우측 Changes 레일을 열고 항목을 차례로 선택합니다.
5. 작은 변경 지점을 선택적으로 확대하고 마지막에는 전체 Diff 관계로 돌아옵니다.

### 장문 AsciiDoc

1. 앱을 새로 실행한 상태에서 합성 `manual.adoc`을 엽니다.
2. Preview 준비 상태와 최초 변환 완료를 가속 없이 보여 줍니다.
3. 목차와 장문 본문이 함께 나타난 결과를 확인합니다.
4. 파일을 닫았다가 다시 열어 캐시된 Preview가 더 빠르게 나타나는 흐름을 보여 줍니다.

### Preview 문맥을 Claude로 전달

1. Preview에서 합성 본문 블록을 클릭해 참고 내용으로 선택합니다.
2. DocPilot의 복사 버튼을 눌러 파일명, 줄 번호와 본문을 실제 클립보드에 넣습니다.
3. Terminal을 열고 실제 `claude` CLI를 실행합니다.
4. 복사한 문맥을 붙여넣고 Enter를 누릅니다.
5. Claude 응답 내용과 응답 시간은 데모의 완료 기준에서 제외합니다.

## 검수 순서

1. 영상을 멈추지 않고 정상 속도로 보며 처음 보는 사용자가 입력, 행동과 결과를 말할 수 있는지 확인합니다.
2. 모든 contact-sheet 프레임을 왼쪽에서 오른쪽으로 확인합니다.
3. 클릭 위치가 확대 또는 포인터 이동에 가려지지 않는지 확인합니다.
4. 마지막 결과가 3초 동안 읽히는지 확인합니다.
5. 매뉴얼 경로를 연속으로 바꿔 각 경로의 영상과 포스터가 즉시 교체되는지 확인합니다.
6. 해상도, fps, 단계 메타데이터, 대체 텍스트와 파일 크기를 검사합니다.
7. 회사명, 사용자 경로, 인증 정보와 관련 없는 창이 없는지 다시 확인합니다.

잘못된 상태를 트리밍이나 속도 변경으로 숨기지 않습니다. 검수에 실패하면 해당 상호작용을 다시 캡처합니다.
